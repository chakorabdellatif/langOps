# LangOps — Architecture Design Document (ADD)

**Status:** Draft v1.0 · **Scope:** MVP (observability only) · **Audience:** contributors implementing LangOps

This document is the implementation blueprint for LangOps: an open-source observability and evaluation platform for LangGraph applications — "Chrome DevTools for LangGraph." It covers the system architecture, monorepo layout, backend design, database schema, SDK instrumentation strategy, frontend architecture, Docker topology, milestones, conventions, and the extension path for post-MVP features.

**MVP constraints (hard):** no authentication, no organizations, no billing, no cloud deployment, no Kubernetes. Everything runs locally via Docker Compose.

---

## 1. Overall Architecture

### 1.1 System overview

LangOps is composed of three deliverable components and three off-the-shelf infrastructure services:

| Component | Type | Responsibility |
|---|---|---|
| `langops-sdk` | Python package (pip: `langops`) | Instruments a LangGraph app in-process; captures node executions, state, prompts, tool calls, LLM metadata; emits OpenTelemetry spans |
| `langops-api` | FastAPI service | Ingests telemetry (OTLP), maps it to the LangOps domain model, persists to PostgreSQL, exposes the query REST API |
| `langops-dashboard` | Next.js app | Visualizes executions, graph DAG, timeline, state diffs, LLM calls, costs, logs |
| OTel Collector | Infra (contrib image) | Receives OTLP from SDKs, batches/retries, forwards to `langops-api` |
| PostgreSQL | Infra | Source of truth for all telemetry data |
| Redis | Infra | Cache for computed aggregates; pub/sub channel for live dashboard updates |

### 1.2 Data flow

Data flows in exactly one direction. There are no callbacks from LangOps into the user's application.

```
┌─────────────────────────────┐
│   User's LangGraph App      │
│   + langops.instrument()    │   in-process, async, non-blocking
└──────────────┬──────────────┘
               │ OTLP/gRPC or OTLP/HTTP (spans + events)
               ▼
┌─────────────────────────────┐
│   OpenTelemetry Collector   │   batching, retry, memory limiting
└──────────────┬──────────────┘
               │ OTLP/HTTP
               ▼
┌─────────────────────────────┐
│   langops-api  (ingestion)  │   parse OTLP → map to domain → persist
│        FastAPI              │
└──────┬───────────────┬──────┘
       │               │ publish "execution updated" events
       ▼               ▼
  PostgreSQL         Redis
       ▲               ▲
       │ queries       │ cached aggregates / pub-sub
┌──────┴───────────────┴──────┐
│   langops-api  (query API)  │   REST, read-only for the dashboard
└──────────────┬──────────────┘
               │ HTTPS/JSON (+ SSE for live updates)
               ▼
┌─────────────────────────────┐
│   langops-dashboard         │
│        Next.js              │
└─────────────────────────────┘
```

### 1.3 Why this shape

- **The SDK speaks standard OTLP, not a proprietary protocol.** Users who already run an OTel pipeline can tee LangOps telemetry into their existing stack (Jaeger, Datadog, Grafana) for free. The LangOps-specific meaning lives in *semantic conventions* (a documented `langops.*` attribute namespace), not in the wire format.
- **The Collector sits between SDK and backend.** It gives us batching, retry, back-pressure, and memory limiting without writing any of that ourselves, and it decouples SDK release cadence from backend release cadence. In local dev the Collector is one container; in a future cloud deployment it becomes a horizontally scalable ingestion tier with zero SDK changes.
- **Ingestion and query are two faces of one service in the MVP.** A single FastAPI app hosts both the OTLP ingestion endpoint and the REST query API. They share the domain and infrastructure layers but have separate presentation modules, so splitting them into two deployables later is a deployment change, not a refactor.
- **PostgreSQL is the only source of truth.** Redis holds only derived, re-computable data (aggregate caches, pub/sub messages). Losing Redis loses nothing.

### 1.4 Communication contracts

| Edge | Protocol | Contract |
|---|---|---|
| SDK → Collector | OTLP (gRPC 4317 / HTTP 4318) | OTel spec + LangOps semantic conventions (`docs/semantic-conventions.md`) |
| Collector → API | OTLP/HTTP → `POST /v1/traces` | Same; backend is a standards-compliant OTLP receiver |
| Dashboard → API | REST/JSON `GET /api/v1/...`, SSE `/api/v1/events` | OpenAPI schema generated by FastAPI; TypeScript types generated from it |
| API → Postgres | SQLAlchemy 2.x async | Alembic-managed schema |
| API → Redis | redis-py async | Cache keys namespaced `langops:cache:*`; pub/sub channel `langops:events` |

### 1.5 Future scalability (summary — details in §10)

The choke points, in the order they will appear, and the pre-planned answer for each:

1. **Ingestion write volume** → split the ingestion module into its own deployable; put Redis Streams (or Kafka) between ingestion and persistence; both sides already communicate through repository interfaces.
2. **Telemetry table growth** → `executions`/`node_executions`/`llm_calls` are keyed by time-ordered UUIDv7 and always queried by `project_id + time`; they can be converted to Postgres native partitions (or TimescaleDB hypertables) without API changes.
3. **Query load** → the query API is stateless and read-only; scale horizontally behind a load balancer, add read replicas.
4. **Multi-tenancy** → `project_id` is already a column on every telemetry table from day one, even though the MVP auto-creates a single default project.

- `examples/` doubles as living documentation and end-to-end smoke tests.
- Nothing in `sdk/` may import from `backend/` or vice versa.

## 3. Backend Architecture (`langops-api`)

### 3.1 Layering

Strict inward-pointing dependencies. A layer may import only from layers below it; the domain imports nothing but the standard library and `datetime`/`uuid`-level primitives.

```
presentation  →  application  →  domain  ←  infrastructure
```

Infrastructure *implements* domain interfaces (dependency inversion): SQLAlchemy repositories satisfy the `Protocol`s declared in `domain/repositories/`. The domain never knows Postgres exists.

### 3.2 Domain layer

**Entities** (plain dataclasses; identity + behavior, no ORM):

- `Project` — namespace for telemetry (MVP: one auto-created "default" project)
- `Graph` — a known graph topology (nodes, edges, version hash)
- `Execution` — one end-to-end graph run: status, timing, thread/checkpoint linkage, token/cost rollups
- `NodeExecution` — one node run within an execution: status, duration, retries, sequence order
- `LlmCall` — one model request: prompts, params, response, token usage, cost
- `ToolCall` — one tool invocation: input, output, error
- `StateSnapshot` — state at a node boundary (input/output) plus computed diff
- `LogRecord` — structured log/error/stack-trace attached to an execution or node

**Value objects** (immutable): `TokenUsage`, `Cost`, `ExecutionStatus` (enum: `running / succeeded / failed / interrupted`), `StateDiff` (added / modified / removed keys), `CheckpointRef` (thread_id, checkpoint_id, parent_checkpoint_id, resumed flag).

**Domain services** (pure functions/classes, fully unit-testable with no I/O):

- `CostCalculator` — token counts + pricing table → `Cost`. Pricing comes in as data; the calculator holds no I/O.
- `StateDiffer` — two JSON-like states → `StateDiff`. (Also used by the SDK conceptually; backend recomputes/validates diffs server-side so the dashboard never depends on SDK version.)
- `ExecutionAggregator` — node/LLM records → execution-level rollups (total tokens, total cost, critical path).

**Repository interfaces** (`typing.Protocol`, async): `ExecutionRepository`, `NodeExecutionRepository`, `LlmCallRepository`, `ToolCallRepository`, `StateSnapshotRepository`, `LogRepository`, `GraphRepository`, `ProjectRepository`, `PricingRepository`. Each exposes intent-revealing methods (`list_recent(project_id, page)`, `get_with_nodes(execution_id)`) — not generic `save/get` CRUD.

### 3.3 Application layer

One class per use case, constructor-injected with the repository interfaces and domain services it needs. The catalog for the MVP:

**Ingestion side:**
- `IngestTelemetryService` — the heart of the backend. Takes parsed OTLP spans, groups them by trace, maps them through `application/mappers/otlp_mapper.py` into domain entities using the semantic conventions, resolves/creates the `Graph` record, computes costs via `CostCalculator`, upserts everything idempotently (spans can arrive twice or out of order — see §3.6), updates execution rollups, and publishes an `execution.updated` event to Redis.

**Query side:**
- `ListExecutionsService` (filter by status/graph/thread/time, paginate)
- `GetExecutionDetailService` (execution + nodes + timeline)
- `GetNodeDetailService` (node + LLM calls + tool calls + state snapshots + logs)
- `GetStateEvolutionService` (ordered snapshots + diffs + context-growth series for one execution)
- `GetGraphTopologyService` (DAG definition + per-node execution overlay)
- `GetCostReportService` / `GetMetricsService` (aggregates; Redis-cached with short TTL)
- `SearchLogsService`

**DTOs** are plain dataclasses returned by services. Presentation converts DTOs to Pydantic response schemas. This keeps the wire format free to evolve (API versioning) without touching use cases.

### 3.4 Presentation layer

- `presentation/api/v1/` — one router per resource: `executions.py`, `nodes.py`, `llm_calls.py`, `tool_calls.py`, `state.py`, `costs.py`, `metrics.py`, `logs.py`, `graphs.py`, `health.py`. Routers do exactly three things: validate input via Pydantic, call one application service, shape the response. No business logic, ever.
- `presentation/ingest/` — the OTLP receiver: `POST /v1/traces` accepting `application/x-protobuf` and `application/json` per the OTLP/HTTP spec. It parses (via `infrastructure/otlp/`) and hands off to `IngestTelemetryService`. Kept in its own module so it can be extracted into a separate deployable later.
- `presentation/events/` — Server-Sent Events endpoint bridging the Redis `langops:events` channel to the dashboard for live execution updates. SSE over WebSockets: one-directional, trivially proxied, no connection state to manage.
- `presentation/schemas/` — Pydantic v2 models for requests/responses only. Never reused as domain objects or ORM models.

**Route map (query API):**

```
GET  /api/v1/executions                      list + filters + pagination
GET  /api/v1/executions/{id}                 detail + node summaries
GET  /api/v1/executions/{id}/timeline        ordered spans for Gantt view
GET  /api/v1/executions/{id}/state           snapshots + diffs + context growth
GET  /api/v1/executions/{id}/logs
GET  /api/v1/nodes/{id}                      node detail (llm calls, tools, state, logs)
GET  /api/v1/graphs                          known graph topologies
GET  /api/v1/graphs/{id}/topology            DAG + execution overlay data
GET  /api/v1/costs/summary                   by model / by graph / by day
GET  /api/v1/metrics/overview                latency percentiles, failure rate, throughput
GET  /api/v1/events                          SSE stream
GET  /api/v1/health                          liveness/readiness
POST /v1/traces                              OTLP ingestion (OTLP-spec path, not under /api)
```

### 3.5 Data model separation (three model families, on purpose)

| Family | Lives in | Purpose |
|---|---|---|
| Domain entities | `domain/entities/` | Business meaning + invariants |
| ORM models | `infrastructure/db/models/` | Persistence shape (tables, indexes, JSONB) |
| API schemas | `presentation/schemas/` | Wire contract with the dashboard |

Mapping between them is explicit (mapper modules). This costs some boilerplate and is the single most deliberate "clean architecture" expense in the design — it is what makes schema migrations, API versioning, and domain refactors independent of each other. It is *not* overengineering here because telemetry schemas evolve fast and the API contract must stay stable for the dashboard and external consumers.

### 3.6 Ingestion semantics: idempotency and ordering

OTLP delivery is at-least-once and spans within a trace arrive in any order (a node span may land before its execution root span). Rules:

- Every table keyed by telemetry uses the **OTel span_id/trace_id as a unique natural key** alongside the UUIDv7 PK; ingestion is upsert-on-conflict, making redelivery harmless.
- An `Execution` row is **created lazily by whichever span arrives first** (child spans carry `langops.execution.id` as an attribute) and enriched when the root span lands.
- Execution rollups (tokens, cost, status) are recomputed from child rows on each relevant upsert, not incremented — idempotent by construction.

### 3.7 Dependency injection & configuration

- **No DI framework.** FastAPI's `Depends` plus one explicit composition root (`composition.py`) that builds the object graph: settings → engine/session factory → repositories → domain services → application services. Routers depend on application services via thin `Depends` providers. Tests build the same graph with fakes — the composition root is the only place that changes.
- **Configuration:** `pydantic-settings` `Settings` class in `infrastructure/settings.py`, sourced from environment variables (12-factor), with `.env` for local dev. Groups: `DATABASE_URL`, `REDIS_URL`, ingestion limits (max payload size, max batch), CORS origins, log level. Fail fast at startup on invalid config.
- **Lifespan:** async engine, Redis pool, and the pricing table cache are created in FastAPI lifespan and disposed on shutdown.

## 4. Database Design (PostgreSQL)

### 4.1 Principles

- **UUIDv7 primary keys** everywhere: time-ordered (index-friendly for time-scoped queries), generatable client-side, safe for future sharding.
- **JSONB for payloads, columns for query dimensions.** Anything we filter/aggregate on (status, model, timestamps, tokens, cost) is a typed column; anything we only display (prompts, state, messages) is JSONB.
- **`project_id` on every telemetry table** from day one (single default project in MVP) — this is the multi-tenancy seam.
- All timestamps `timestamptz` (UTC). Costs as `numeric(12,6)` USD.

### 4.3 Relationships

```
projects 1──n graphs
projects 1──n executions
graphs   1──n executions
executions 1──n node_executions ──n llm_calls
                                 └─n tool_calls
executions 1──n state_snapshots (optionally → node_executions)

```

Deletes cascade down from `executions` so retention cleanup is a single delete per run. No metrics table in the MVP: metrics are computed from telemetry tables at query time and cached in Redis; a pre-aggregated rollup table is a documented future optimization, added only when query latency demands it.

## 5. SDK Architecture (`langops-sdk`)

### 5.1 Design goals

Zero-code-change adoption (`instrument(graph)` and done), zero blocking of the user's app (all export is batched and async), graceful degradation (telemetry failures must never break the user's graph), and standards compliance (plain OTel spans that any OTLP backend can receive).

### 5.2 Public surface

The public API is intentionally tiny: `instrument(compiled_graph, config: LangOpsConfig | None) -> CompiledGraph` and `LangOpsConfig` (endpoint, service/graph name, capture toggles, `max_payload_bytes`, redaction hook, sampling ratio). Everything else is private.

### 5.3 The three hook points inside LangGraph

LangGraph is built on the LangChain Runnable protocol and exposes three natural, *supported* seams. We use all three and monkeypatch nothing:

1. **Graph wrapper** (`instrumentation/graph.py`) — `instrument()` wraps the `CompiledGraph`'s entrypoints (`invoke`, `ainvoke`, `stream`, `astream`). The wrapper: opens the **execution root span**; records graph input/output; extracts `thread_id` / `checkpoint_id` from the run `config["configurable"]`; captures the topology once via `graph.get_graph()` (nodes + edges → `langops.graph.topology` attribute); and injects the callback handler (below) into the run config. It determines fresh-vs-resumed by asking the (wrapped) checkpointer whether a checkpoint already exists for the thread before the run starts.

2. **Callback handler** (`instrumentation/callbacks.py`) — a `BaseCallbackHandler` injected into the run config, receiving LangChain/LangGraph runtime events:
   - `on_chain_start/end/error` for graph *nodes* → **node spans** (name, sequence, timing, status, error+stack, retry detection via repeated starts of the same node/task id). Node input/output payloads give per-node state before/after.
   - `on_chat_model_start` / `on_llm_end/error` → **LLM call spans** (model, provider, messages incl. system prompt, params, response, and token usage from the response's `usage_metadata`).
   - `on_tool_start/end/error` → **tool call spans**.
   Parent-child wiring uses the handler's `run_id`/`parent_run_id` pairs mapped onto OTel span context, so the span tree mirrors the actual execution tree (including subgraphs).

3. **Checkpointer wrapper** (`instrumentation/checkpointer.py`) — if the graph has a checkpointer, `instrument()` wraps it (composition: a decorator object implementing `BaseCheckpointSaver`, delegating to the original). Every `put()` is the authoritative moment state is persisted: the wrapper captures the checkpoint id, parent checkpoint id, channel values (the state), and emits a **state snapshot event** with a structural diff against the previously seen state. This is more reliable than inferring state from node inputs/outputs and is exactly where thread/checkpoint metadata lives. Graphs without a checkpointer degrade gracefully to callback-derived state capture.

### 5.4 From events to telemetry

Internal pipeline: **capture → semantic mapping → export**.

- **Capture** normalizes raw LangGraph/LangChain event payloads: safe serialization of arbitrary state (fallback `repr`, depth limits), size capping at `max_payload_bytes` with a `langops.truncated` marker, structural diffing (`capture/diff.py`), and the user's redaction hook applied *before* anything reaches the exporter (secrets never leave the process).
- **Semantic mapping** assigns every span a `langops.kind` (`execution | node | llm | tool`) and namespaced attributes per `docs/semantic-conventions.md` — e.g. `langops.node.name`, `langops.node.sequence`, `langops.llm.model`, `langops.llm.input_tokens`, `langops.state.diff`, `langops.checkpoint.id`, `langops.thread.id`. Where official OTel GenAI semantic conventions exist (`gen_ai.*` for model, tokens, prompts) we emit those names and add `langops.*` only for LangGraph-specific concepts (state, checkpoints, topology, sequence). Large payloads (state snapshots, full messages) ride as **span events**, not attributes, to respect attribute size norms.
- **Export** is stock OpenTelemetry: a dedicated `TracerProvider` (never hijacking the user's global provider — coexistence with existing instrumentation), resource attributes (`service.name`, `langops.sdk.version`, project), `BatchSpanProcessor` for async batched export, and the standard OTLP exporter pointed at the Collector (`LANGOPS_ENDPOINT`, default `http://localhost:4317`).

**Failure policy:** every capture path is wrapped so that an internal SDK error logs one warning and drops that datum; the user's graph run is never failed or slowed synchronously by LangOps.

### 5.5 Why OpenTelemetry rather than a custom protocol

Batching, retry, back-pressure, sampling, context propagation, and wire format are solved problems in OTel; reimplementing them is pure liability. The trade-off — OTLP attribute/event size conventions constrain giant payloads — is handled by explicit payload limits and span events, and is revisited in §10 (a side-channel for oversized artifacts is a documented future extension that changes only the SDK export layer and the ingestion mapper).

### 6.3 State management

|---|---|---|
| Server data (executions, nodes, costs…) | **TanStack Query** | caching, pagination, revalidation, request dedupe |
| Live updates | SSE → invalidate/patch TanStack Query caches | one mechanism for freshness; no second data path |

### 7.2 Networking

- One bridge network, `langops`. Inside it, services address each other by service name (`postgres:5432`, `redis:6379`, `api:8000`) — hence `DATABASE_URL` uses host `postgres`, and the Collector exports to `http://api:8000/v1/traces`.
- Only four ports are published to the host, and each has a distinct audience: **3000** dashboard (browser), **8000** API (browser fetches — the dashboard runs client-side, so the API must be host-reachable, which is why `NEXT_PUBLIC_API_URL` is `localhost:8000` not `api:8000`), **4317/4318** Collector (the user's instrumented LangGraph app runs on the *host*, outside compose, and sends OTLP to `localhost:4317`). Postgres/Redis ports are published purely for developer inspection and can be removed.

### 7.3 Volumes

- `postgres_data` named volume — the only persistent state in the system. `docker compose down -v` is a documented full reset.
- The Collector config is a read-only bind mount so it can be edited without rebuilding an image.
- Redis is deliberately unpersisted: cache and pub/sub only (§1.3).

### 7.4 Environment variables

Single `.env` at repo root (from `.env.example`), interpolated by Compose. Rules: every variable has a working local default; the backend reads *only* env (never files) via `pydantic-settings`; `NEXT_PUBLIC_*` variables are baked into the dashboard at build time and documented as such; secrets in `.env` are acceptable for the local-only MVP and the file is gitignored.

## 9. Coding Conventions

**Naming.**
Python: `snake_case` modules/functions, `PascalCase` classes, `UPPER_SNAKE` constants; application services named `<Verb><Noun>Service` (`GetExecutionDetailService`); repository Protocols `<Entity>Repository`, implementations `Postgres<Entity>Repository`. TypeScript: `PascalCase` components/types, `camelCase` functions/vars, hooks `use<Thing>`, feature folders `kebab-case`. SQL: plural `snake_case` tables, singular columns, FKs `<entity>_id`. OTel attributes: dot-namespaced per `semantic-conventions.md`; never invent an attribute name inline — add it to the conventions doc and constants first.

**Logging.**
Structured JSON logs (structlog or equivalent) with `event`, `trace_id`/`execution_id` correlation fields. Levels: DEBUG dev detail, INFO lifecycle events, WARNING degraded-but-working, ERROR failed operation. Never log prompt/state payload contents at INFO or above (they are user data); the SDK logs at most one WARNING per failure class.

**Errors.**
Backend: exception hierarchy rooted at `LangOpsError`; domain raises domain errors (`ExecutionNotFound`), presentation maps them to HTTP via exception handlers (404/422/500) with a uniform error body `{code, message, detail}`. Never catch-and-swallow; catch-map or let it propagate. SDK: the inverse policy — catch everything at the instrumentation boundary, warn once, drop the datum (§5.4). Ingestion: reject malformed OTLP with 400; never 500 on bad input.

**Folder organization.**
Layered rules of §3 are enforced with import-linter contracts in CI (domain imports nothing framework-y; presentation never imports infrastructure directly). Frontend: `app/` routes stay thin; feature code lives in `features/<name>`; promotion to `components/` requires a second consumer.

**Testing.**
Pyramid: unit (domain/application with in-memory fakes of the repository Protocols — no mocking of SQLAlchemy internals), integration (real Postgres via testcontainers for repositories and ingestion), API (httpx against the ASGI app), SDK golden-file span tests + fault-injection, E2E (compose + example apps, M8). Naming `test_<unit>_<scenario>_<expectation>`. Coverage gate on domain + application layers (target ≥ 90% there, not repo-wide vanity coverage).

**Commit style.**
Conventional Commits: `feat(sdk): ...`, `fix(api): ...`, `docs:`, `chore:`, `refactor:`, `test:`; scopes = `sdk | api | dashboard | collector | docker | docs`. PRs squash-merge; PR title becomes the commit. Breaking changes to the semantic conventions require a `feat(semconv)!:` commit plus a version bump of the conventions doc.

**Architecture principles.**
Dependencies point inward; composition over inheritance (the checkpointer wrapper, not a subclass, is the canonical example); interfaces (`Protocol`) only where a second implementation is plausible or needed for testing — repositories yes, one-off helpers no; every significant decision gets an ADR in `docs/adr/`; when in doubt, choose the boring solution and leave a seam.

## 10. Future Roadmap — Extending Without Refactoring

Each planned feature maps onto a seam that already exists in this design:

- **Prompt management & versioning.** New domain entities (`Prompt`, `PromptVersion`) + tables + a write API — a purely *additive* vertical slice through the existing layers. LLM calls already store full request messages, so back-linking a call to a prompt version is a nullable FK and a fingerprint match on ingest. No existing table changes shape.

- **Agent evaluation.** Evaluations are consumers of data the MVP already persists (inputs, outputs, state, costs). Add `evaluations` / `evaluation_runs` tables, an `EvaluationService` in the application layer, and a worker process that reuses the existing infrastructure layer. Redis is already present to serve as the job queue (or is swapped for a real queue behind a `TaskQueue` Protocol).

- **Execution replay & time travel.** The hard part — capturing thread/checkpoint lineage and full state snapshots — is in the MVP schema from day one (`checkpoint_id`, `parent_checkpoint_id`, `state_snapshots`). Replay is a new application service that reconstructs a `RunnableConfig` at a chosen checkpoint and re-invokes the user's graph via the SDK; time-travel *visualization* is a dashboard feature over data we already store.

- **Human-in-the-loop.** LangGraph interrupts already surface as `interrupted` execution status. HITL adds a write path (resume/approve endpoints) and a pending-interrupts view. The SSE channel already delivers the live "waiting for input" signal.

- **Multiple projects.** `project_id` is on every table and every repository method signature already. The work is CRUD for projects, a project switcher in the dashboard, and an SDK config field — no data migration.

- **Multiple users / auth.** Auth was excluded, not precluded: it enters as (a) a FastAPI dependency at the presentation layer (API keys per project → later OIDC), and (b) `users` / `api_keys` tables. Domain and application layers gain an optional `actor` on write use cases. Because presentation is the only HTTP-aware layer, auth touches exactly one layer.

- **Kubernetes / cloud deployment.** Every service is already a stateless 12-factor container (state lives only in Postgres); ingestion and query are separable modules within the API (§1.3); the Collector is the standard cloud-native ingestion tier. The path: Helm chart, managed Postgres/Redis, HPA on the query API, a queue between ingestion and persistence when write volume demands it (§1.5). No application code changes — deployment artifacts only.

- **Scale-out storage.** Time-ordered UUIDv7 keys + time-scoped query patterns mean the telemetry tables convert to native partitions or TimescaleDB without API changes; heavy payloads (state, messages) can move to object storage behind the existing repository interfaces, with JSONB columns becoming pointers.

The common thread: the MVP spends its complexity budget on **boundaries** (semantic conventions, repository interfaces, layer separation, `project_id` everywhere) and avoids spending it on **speculative features**. That is what keeps each future feature additive.
